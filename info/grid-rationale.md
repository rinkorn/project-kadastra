# Регулярная сетка в кадастровой оценке — методология

Выжимка доклада о применении регулярной гексагональной сетки для расчёта ценообразующих факторов (ЦОФ) и моделирования рынка недвижимости (контекст Татарстана / ГБУ, опыт туров 2022–2023 гг.). Документ описывает рациональ сетки, её параметры, способы расчёта ЦОФ на сетке, способы моделирования и перспективы масштабирования на НСПД.

## 1. Контекст и цели

- **Не кадастр, а модель рынка.** Подход выходит за рамки методических указаний по кадастровой оценке: первичная задача — построение **модели рынка недвижимости**. Кадастровая стоимость — частный случай её применения.
- **Без ML делать нечего.** Рынок — сложная нелинейная динамическая система. Линейные методы и метод средних не моделируют его адекватно.
- **Исключение субъективности.** Сетка нужна, чтобы группировка и расчёт ЦОФ были детерминированы: одни и те же объекты всегда обрабатываются одинаково, эксперт не вносит свою «экспертизу» в результат.
- **Масштабируемость.** При переходе на двухгодичный цикл объём работ ГБУ ×2; на сетке это масштабируется без потери времени и качества (19 000 vs 200 000 объектов в день — обрабатываются одинаково).

## 2. Зачем регулярная сетка — выгоды для кадастровой оценки

1. **Распараллеливание треков работ.** ЦОФ считаются на всей территории, а не по объектам оценки. Группировка, анализ рынка и расчёт ЦОФ становятся независимыми треками.
2. **ЦОФ, которых иначе не посчитать.** Часть факторов в принципе невозможно или крайне трудоёмко получить без сетки.
3. **Относительные ЦОФ.** Сетка даёт распределение значений на территории → можно конструировать большое количество производных признаков, которые объясняют вариацию цен лучше «сырых» расстояний (подтверждено отчётом за 2023).
4. **Субъектовые модели вместо 28 сегментных.** Одна модель на регион способна объяснять цены и в крупных городах, и в мелких НП — за счёт богатого пространства относительных ЦОФ.
5. **Репрезентативность за пределами объектов оценки.** Слабое место текущей практики ГБУ: «репрезентативность» сводится к мин/макс. Корректная репрезентативность — это совпадение распределений значений ЦОФ на выборке и генеральной совокупности.
6. **Скорость дооценки.** В тур 2023 на Python-конвейере: земля — ~3 рабочих дня, ОКС — ~5 (главный тормоз — определение кадастровой стоимости сооружений, пока не автоматизировано).
7. **Точнее для крупных объектов.** Если объект больше ячейки, оценка идёт по каждому покрывающему гексагону отдельно (часть участка у дороги — дороже, дальняя — дешевле). Сумма по ячейкам ≠ один расчёт ЦОФ для участка целиком; разница может составлять заметные суммы налогов.

## 3. Сетка в контексте НСПД (масштабирование на РФ)

- **Унификация ЦОФ.** Главное условие единого реестра — не унификация названий, а **унификация способа расчёта**. Сейчас в НСПД часто только «расстояние по прямой / вхождение в зону». На практике способов ~10+. Без согласованного способа расчёта значения ЦОФ из разных регионов несравнимы.
- **Единый реестр как конструктор.** Перечень типов точек интереса (школы, садики, магазины и т.д.) ограничен → реализуем как библиотеку конструктивных способов расчёта, а не как набор готовых слоёв.
- **Межсубъектовые модели.** Без унифицированных ЦОФ нельзя расширять обучающую выборку соседними регионами: чужие сделки требуют пересчёта ЦОФ «как у себя». С единым способом расчёта межсубъектовые модели становятся технически возможными.
- **Темпоральная сетка для моделей вглубь по времени.** Если сетка стабильна между турами, можно хранить срезы ЦОФ по годам. Аналог трёхлетней давности → берём значения ЦОФ той даты (например, метро ещё не было). Без сетки — крайне сложно.
- **Снижение затрат на хранение в АДГКО.** Хранить значения ЦОФ для каждого объекта смысла нет (оценка раз в 2–4 года). Дешевле считать на лету при разъяснениях, имея сетку и формулу. Особенно при росте числа ЦОФ (модель 2023 — 55 ЦОФ; ожидание на следующий тур — до 10 000).
- **Перманентная кадастровая оценка** на сетке реализуется естественно.
- **Анализ «что если».** Поставили ещё одну школу → пересчитали ЦОФ → увидели изменение стоимости вокруг. Применимо и к рынку, и к распределению бюджетных средств.
- **Визуализация ценовой поверхности.** На каждый гексагон можно посчитать УПКС по любой модели → карты для инвесторов / публичных сервисов.

## 4. Параметры сетки

- **Гексагон 100 м (плоский, лежащий на боку).** На территории Татарстана — около 7 млн ячеек. 200-метровый — 18 млн (избыточно).
- Площадь одной ячейки ~8 600 м². Для сравнения: Московский Кремль ≈ 4 таких гексагона, Казанский ≈ 27.
- Сельские НП считаются как городские (в туре 2022 это уже использовалось — без упрощений «один НП = один удельник»).
- Возможна **переменная детализация**: ячейки разного размера в разных зонах (см. ниже про разрежение опорных точек).
- Площадь Татарстана ≈ 6% территории РФ ≈ суммарная площадь всех населённых пунктов РФ → расчёт реалистичен и в масштабах страны на одной машине за ночь.

### Почему гексагон, а не квадрат

- Меньше артефактов на границах при интерполяции.
- Минимальное отношение периметра к площади.
- В контексте «сетка как инфраструктура для расчётов» это вкусовщина — можно и квадрат. Гексагон субъективно лучше и красивее.

## 5. Классификация ЦОФ

### 5.1. По смыслу (методические указания)

| Группа | Свойства |
| --- | --- |
| Характеристики объекта | присущи самому объекту оценки |
| Характеристики окружения | стационарные во времени, школы/магазины меняются медленно — допустимо использовать срезы как константы |
| Характеристики среды | могут быть нестационарны во времени; критичны при использовании аналогов более чем за 1–2 года |

### 5.2. По способу расчёта (5 групп)

1. **Объектные** — присваиваются по кадастровому номеру или вычисляются по геометрии объекта (площадь, аварийность, категория земель, ориентация в пространстве относительно дорог, рельеф).
2. **Дистанционные** — расстояния до точек интереса.
3. **Зональные** — площадь / плотность в буферной зоне.
4. **Территориальные** — присвоение значения по принадлежности к зоне (муниципалитет, видовое место, опорный НП и т.п.).
5. **Временны́е** — на дату оценки одинаковы для всех; используются для аналогов из прошлого.

## 6. Объектные ЦОФ

- Простое присваивание по кадастровому номеру / по графике.
- Площадь, категория земель, аварийность.
- **Ориентация объекта относительно дорог** — при равной площади у вытянутых вдоль/поперёк улицы участков разная стоимость. Считается по геометрии.
- **Рельеф и крутизна склона** — для специальных случаев (огороды, сельхозугодья). На сетке считать неудобно — проще объектно.

## 7. Дистанционные ЦОФ

### 7.1. Опорные точки (разрежение сетки)

- Считать расстояния через каждые 100 м на межселенке бессмысленно.
- В туре 2023 использовали **разрежённую сетку**: 100 м внутри НП, 200 м в буферных зонах вокруг крупных дорог и НП, 500 м в межселенке. ~7–8 млн → ~1.3 млн опорных точек.
- Разница между 100-м и 200-м сеткой при пилотных замерах — десятые доли процента.
- В Python-конвейере (без QGIS) скорость выросла на 3–4 порядка → разрежение становится менее принципиальным.

### 7.2. Граф дорог

- Используется **граф OSM** (вариант 2022, в т.ч. потому что новых трасс с тех пор было немного).
- Свой граф пробовали — поддержка муторная; OSM обновляется быстрее и точнее (проверено сравнением).
- Покрытие — Татарстан + буфер (есть точки, до которых из РТ нельзя доехать, не выехав за её пределы).
- Граф используется с **путевым временем** (вес ребра — не евклидово расстояние, а с учётом скорости/класса дороги). Ближайший = ближайший по времени, не по геометрии.

### 7.3. Варианты способа расчёта (важно для НСПД)

В туре 2023 (реализовано):

1. Расстояние по прямой.
2. Расстояние до ближайшей точки интереса по графу (путевое время).
3. Среднее расстояние до **N ближайших** точек интереса (комплексно учитывает и доступность, и плотность).

В разработке:

1. **Эмуляция графа без графа дорог** на основе модели рельефа (SRTM / FABDEM): строим граф по DEM, штрафуем рёбра по перепаду высот. Дороги — это плоские места; сильные перепады отсекаются. На дальних дистанциях расстояния близки к расчёту по реальному графу. Сильно дешевле в поддержке.
2. **Транспортное моделирование.** Используем тот факт, что мы знаем расположение и тип всех зданий (жилое/нежилое, школа/работа/торговля). Моделируем утренний/вечерний поток людей → находим узкие места графа, где «реальное» время доезда сильно отличается от номинального. Для агломераций от ~0.5 млн жителей это сильно объясняет различия цен.

«Индустриальный default» НСПД:

1. **Вхождение в зону / процент покрытия** — сейчас фактически это всё, что используется в НСПД.

### 7.4. Интерполяция значений с опорных точек на сетку

- Растровая интерполяция всей территории Татарстана даёт ~15 ГБ — неудобно.
- Татарстан разрезан Волгой/Камой/водохранилищами → интерполяция через водные преграды бессмысленна, на краях артефакты.
- **Решение:** интерполяция в крупных гексагонах 5 км → значение в каждом 100-м гексагоне берётся как **медиана** по родительскому 5-км. Медиана устойчивее к огрехам, чем среднее.

## 8. Зональные ЦОФ

- **Моно-площадь.** В буфере вокруг гексагона считается доля одного слоя (например, водная гладь).
- **Поли-площадь.** Несколько слоёв одновременно, доли суммируются (могут дать > 1). Учитывает кумулятивный эффект негатива: «гусиная ферма + завод» ≠ «один негативный объект». В моделях 2023 поли-площадь объясняла лучше, чем моно-площадь.
- **Плотность точек в буфере.** Радиусы 100/300/500/800 м. Можно использовать число точек, а можно — среднее расстояние до N ближайших (см. дистанционные).

Зональные ЦОФ важны для крупных объектов: при оценке участка «целиком» лучшие куски усредняются с худшими, а на сетке каждая ячейка считается отдельно.

## 9. Территориальные ЦОФ

- Присваивание признаков по принадлежности гексагона к территории.
- Сетка заранее разрезана по муниципалитетам → не нужен отдельный пространственный запрос для присвоения макроэкономических показателей муниципалитета. Один пространственный запрос вместо двух.
- Сюда же ложится та субъективная часть, которую невозможно убрать: «видовые места», «престижные кварталы», «опорные НП», «вход/не вход в НП» — как комбинационно-переменные.
- При масштабировании НСПД — предрасчёт по границам муниципальных образований, чтобы не таскать сетку целиком.

## 10. Временны́е ЦОФ

- На дату оценки одинаковы для всех объектов оценки → нулевая дисперсия → в саму модель не входят.
- Нужны для **корректного использования аналогов из прошлого**: на дату аналога эти показатели были другими. Без темпоральных срезов сетки аналог 3-летней давности «врёт» по расстояниям (метро ещё не построено и т.п.).

## 11. Конструирование признаков на сетке (относительные ЦОФ)

Главный выигрыш сетки для моделирования: распределение значений ЦОФ по территории известно полностью (а не только на объектах оценки). Это позволяет генерировать большое число производных признаков:

- **Стандартизация / нормализация ЦОФ** относительно разных территорий: субъект, муниципальный район, муниципальное образование, НП, градостроительный квартал, буферная зона, соседи 1/2/3 порядка от объекта.
  - Каждый дополнительный способ нормализации удесетеряет число признаков.
  - Перебор — алгоритмический (не вручную); проблема размерности и шум — отдельные задачи (кросс-валидация, отбор).
- **Функции влияния** — нелинейные преобразования (в туре 2023 — монотонные, по-хорошему должны быть и немонотонные).
- **Взаимодействия признаков** — произведения вида «расстояние до центра × расстояние до школы». Усложняет модель, но работает; ни один из ГБУ пока не использует.
- **Дискретизация (бинирование)** количественных в качественные — наивный «есть в 500 м / нет». Граница — экспертная, поэтому стараемся не использовать.
- **Признаки из символьной регрессии** — модель без ограничений подгоняется к данным; из неё извлекаются формы, далее используются как признаки.

Контекст: на 2023 у моделей ~55 признаков. На следующий тур планируется до 10 000. В аэродинамике подобные задачи решаются на ~миллионе признаков — это не «много», это норма для современного ML.

## 12. Применение ЦОФ к объектам оценки

После того как для каждого гексагона посчитаны все ЦОФ:

1. Один пространственный запрос — пересечение объекта оценки с сеткой.
2. Внутри пересечения — взвешивание по доле перекрытия.
3. На вход модели уходит готовый вектор признаков.

В туре 2023 «наивная» реализация экспликации в QGIS укладывалась в ~3.5 часа на всю землю РТ; вся дооценка по земле — ~4 часа. На текущем Python-стэке — кратно быстрее.

## 13. Моделирование

### 13.1. Тур 2023 (что было)

- **MLP** (3 слоя × 512 нейронов, ReLU) — для **детектирования выбросов**, не для классификации/регрессии. Аномалии не выбрасываются, а маркируются (вверх/вниз) и остаются в обучении.
- **Gradient Boosting Regressor** (`sklearn`) с **Optuna** для гиперпараметров.
- ~55 признаков, одна субъектовая модель на регион.

### 13.2. Следующий тур — 4 параллельные модели

| Модель | Назначение |
| --- | --- |
| **Black Box** | Сложная неинтерпретируемая модель — верхняя планка качества |
| **Grey Box** | Аппроксимация Black Box до сложных функций (без жёсткой монотонности) |
| **White Box** | Интерпретируемая модель (как в 2023) — то, что фактически сдаётся |
| **Наивная линейная регрессия** | С перебором знаков при регрессорах — нижняя планка |

Цель — измерять **потери при упрощении**: насколько White Box хуже Black Box, насколько подозрительна жалоба заявителя в свете обеих моделей. Простая модель не аппроксимирует сложную без потерь — теряются налоги (не столько в сумме, сколько в перераспределении: дешёвые становятся дороже, дорогие — дешевле).

### 13.3. Инструментальный контекст

- LLM-ассистенты (Qwen, GLM, Gemini и др.) кратно ускоряют разработку и эксперименты. То, что в 2023 писалось вручную, теперь — небо и земля.

## 14. Инфраструктура и реализация

### 14.1. Железо

- **Одна локальная машина**, никаких серверов. Все расчёты по туру (земля + ОКС) выполняются на ней.
- Конфигурация: Ryzen 7 5800X (8 ядер), 128 GB RAM, ~6 TB дискового пространства. Видеокарта — для локальных LLM.
- На наши задачи этого хватает с запасом, в т.ч. при «индийском коде». Объём базы по дооценке — ~20 GB.

### 14.2. Тур 2023 (QGIS + Python)

- Один Python-скрипт + **графическая модель в QGIS Processing Modeler**, где зашита вся последовательность: «возьми X → проверь Y → выгрузи Z».
- В середине — несколько кнопок, которые оператор нажимает вручную (между ними «успеваешь налить кофе»).
- Скорость не самоцель: дооценка за 1, 10 или 15 минут — без разницы. Микросервисы и low-latency не нужны.
- Стиль — наивное процедурное программирование: задача → подзадачи → реализация каждой → тесты на стыках, чтобы ничего не «уплыло» между шагами.

### 14.3. Текущий стек (2024+)

- **QGIS выкинут полностью.** Всё в коде, никакого UI: одна точка входа, лог в текстовый файл, состояние — в локальной PostgreSQL (БД небольшая, ~20 GB; устроена «супернаивно», но под задачи хватает).
- Стек: Python stdlib + **Fiona**, **GDAL**. Никаких плагинов и оболочек.
- Логика переноса: QGIS — это оболочка над Python-интерпретатором, все его модули сами на Python (или биндинги к C). Всё, что делал QGIS, переписывается на стандартные библиотеки 1:1.
- Скорость на чистом Python — ориентировочно на 3–4 порядка быстрее, чем тот же расчёт через QGIS-модели.

### 14.4. LLM-ассистенты в разработке

- **Локальные LLM-кодеры** (Qwen, GLM и др.) запускаются на той же машине, без облака.
- Цикл: открыл редактор → дал ТЗ модели → получил код → запустил в Spyder/Jupyter → проверил вывод. Перекладка нового расчёта с QGIS на чистый Python — буквально за день.
- **Ключ к валидации** — наличие эталонного результата из тура 2023 (тот же расчёт, выполненный вручную в QGIS). Сравниваешь: совпало → идёшь дальше. Без эталона vibe-coding опасен; здесь он работает, потому что ожидаемый ответ заранее известен.

## 15. Перспектива: сверточные сети и geometr-матрицы

Будущий шаг — отказ от «вектор чисел на вход» в пользу **изображений ЦОФ**:

- На вход подаётся не одно расстояние, а **картинка** значений ЦОФ вокруг объекта (например, 512×512 пикселей; на одну сделку — порядка 16 тыс. часов на признаках на разных сечениях).
- Один пиксель = бинарный признак (0/1: суша/океан, есть/нет школа и т.п.) или непрерывное значение ЦОФ.
- На уровне субъекта данных не хватает (мало аналогов), но на уровне РФ или нескольких субъектов — хватает.

Ограничение, которое формирует всю текущую архитектуру: методические указания требуют **интерпретируемой формулы**. Поэтому из неинтерпретируемой Black Box модели приходится «выкручивать» интерпретируемую White Box ценой богатого пространства относительных ЦОФ. Без этого ограничения работали бы прямо с Black Box / CNN.

## 17. Ключевые ограничения и риски

- **Репрезентативность выборки.** Большинство существующих моделей ГБУ невалидны по этому критерию (мин/макс ≠ совпадение распределений). Применять страшно.
- **Нелинейность рынка.** Любая модель, аппроксимирующая Black Box до простой формулы, теряет в точности и налогах. Нужно явно мерить эту потерю.
- **Графовые расчёты дорогие.** Делать кратчайший путь для каждого объекта оценки в каждом туре нереально → расчёт «всё на сетке один раз» — единственный путь к автоматизации.
- **Хранение ЦОФ для каждого объекта** при росте числа ЦОФ нерационально. Считать на лету по запросу.
- **НСПД без унификации способов расчёта** ЦОФ — единый реестр работать не будет.
