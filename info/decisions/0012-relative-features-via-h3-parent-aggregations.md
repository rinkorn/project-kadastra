# ADR-0012: Относительные ЦОФ через агрегаты на родительских H3-гексах

**Статус:** Accepted
**Дата:** 2026-04-25
**Реализует:** [ADR-0010](0010-methodology-compliance-roadmap.md), пункт 2 дорожной карты трека 2 (методологическое соответствие).
**Источник методологии:** [info/grid-rationale.md](../grid-rationale.md), §11.

## Контекст

До этого ADR модель видела ЦОФ объекта только в абсолютных значениях: «расстояние до метро = 2 800 м», «этажность = 9», «год постройки = 1985». Методология (§11) говорит, что **главный заявленный выигрыш сетки** — генерация **относительных** признаков, отвечающих на вопрос «насколько объект отличается от своего окружения».

> [grid-rationale.md §11] Главный выигрыш сетки для моделирования: распределение значений ЦОФ по территории известно полностью (а не только на объектах оценки). Это позволяет генерировать большое число производных признаков. Стандартизация / нормализация ЦОФ относительно разных территорий: субъект, муниципальный район, муниципальное образование, НП, градостроительный квартал, буферная зона.

«Расстояние до метро 2 800 м» в центре Казани и на окраине агломерации — две принципиально разные ситуации, и модель этого не понимает, пока ей не показать «а медиана по району — 1 100 м» (центр) против «медиана по району — 6 200 м» (окраина). С относительной фичей оба объекта получают одинаковую информацию: «у меня дальше, чем у соседей». Без неё модель пытается выучить эту разницу через комбинации lat/lon, что и менее эффективно, и менее переносимо при перевыборках территорий.

## Решение

**Для каждого числового ЦОФ строить производные признаки относительно агрегатов на родительских H3-гексах.**

### Иерархия агрегации

Используем родительские разрешения **`[7, 8]`** ([ADR-0007](0007-kazan-agglomeration-scope.md)):

| res | средняя сторона | средняя площадь | роль |
|-----|------------------|------------------|---------|
| 7   | ~1.4 км          | ~5 км²           | район агломерации (контекст) |
| 8   | ~530 м           | ~0.7 км²         | микрорайон / квартал |

Res=9 и глубже не берём в первой итерации: на res=9 (~0.1 км²) родительская группа часто состоит из 1–3 объектов, агрегаты вырождаются в «сам объект ± шум». Имеет смысл вернуться к ним с поли-площадными буферами (блок 3 трека 2).

«Субъект / муниципалитет / НП» из методологии (§11) пока не подключаем — это блок 4 (территориальные ЦОФ через ГАР), отдельным ADR.

### Базовые агрегаты

Для каждого (числовой ЦОФ × parent_res) считаем:

- `count` — число объектов в родительском гексе (число, по которому модель может «дисконтировать» агрегат — медиана из 1 объекта ≠ медиана из 50);
- `median` — устойчивая средняя;
- `p25`, `p75` — квартили (для оценки разброса родителя без явного std).

Сразу не считаем `mean`/`std`: на скошенных распределениях (расстояния, годы постройки) они менее информативны, а CatBoost получит их через медиану + квартили в любом случае.

### Производные признаки

Для каждого исходного ЦОФ `F` и каждого parent_res `R`:

| Имя | Формула | Что отвечает |
|-----|---------|--------------|
| `F__rel_p{R}_diff_med` | `F − median_p{R}` | «насколько я больше/меньше типичного соседа» (в исходных единицах) |
| `F__rel_p{R}_ratio_med` | `F / median_p{R}` | «во сколько раз я больше/меньше» (масштабоинвариантно) |
| `F__rel_p{R}_z_iqr` | `(F − median_p{R}) / (p75_p{R} − p25_p{R})` | робастный z-score (для CatBoost опционально, но полезен в White Box модели) |
| `count_p{R}` | размер родителя | контекст для модели — «насколько надёжен агрегат» |

`ratio_med` и `z_iqr` могут давать `NaN` (median = 0; p75 = p25). NaN пропускаем в обе стороны — CatBoost handles missing nativel'но (см. ADR-0010 раздел про NaN handling).

### Архитектура

Чистая ETL-функция (без порта — нет внешних зависимостей):

```text
kadastra/etl/relative_features.py
└── compute_relative_features(
        objects: pl.DataFrame,
        *,
        parent_resolutions: list[int],
        feature_columns: list[str],
    ) -> pl.DataFrame
```

Контракты:

- `objects` обязан иметь колонки `lat: Float64`, `lon: Float64` и каждую `feature_columns[i]` (Int/Float).
- На выходе те же колонки + `parent_h3_p{R}: UInt64` для каждого `R` + четвёрка производных (`diff_med`, `ratio_med`, `z_iqr`, `count_p{R}`) для каждой пары `(F, R)`.
- Агрегаты считаются на текущем переданном датафрейме — **по объектам Казанской агломерации**. Это то, что в методологии называется «по объектам оценки», а не по сетке всех гексов: для пилотного датасета достаточно. Когда подключим сделочные данные, агрегаты будут считаться по полному пулу сделок (не только по обучающей выборке) — отдельным ADR.

`BuildObjectFeatures.execute()` зовёт `compute_relative_features` **последним шагом** — после metro/road/neighbor/synthetic target. Так в `feature_columns` уже видны и графовые расстояния, и сетевые счётчики, и атрибуты NSPD.

### Settings

```python
relative_feature_parent_resolutions: list[int] = [7, 8]
relative_feature_columns: list[str] = [
    "dist_metro_m",
    "dist_entrance_m",
    "count_stations_1km",
    "count_entrances_500m",
    "levels",
    "year_built",
    "area_m2",
    "road_length_500m",
]
```

(Точный набор `feature_columns` подберём по тому, что фактически живёт в gold-фрейме после `compute_object_road_features` / `compute_object_neighbor_features`.)

### Per-class или single?

Агрегаты считаем **поверх всех классов** (apartment + house + commercial + landplot вместе). Логика: «соседство» в методологии — это территория, а не набор однотипных объектов; на res=8-гексе живёт смесь жилья, магазинов и участков, и это и есть тот контекст, относительно которого нормируется. Если потом окажется, что нужны class-conditional агрегаты — отдельным ADR.

## Последствия

- Новая чистая функция `compute_relative_features` + unit-тесты.
- `BuildObjectFeatures.execute()` дополняется одним шагом.
- `Settings` получает два новых параметра (`relative_feature_parent_resolutions`, `relative_feature_columns`).
- Объём фичей растёт **аддитивно**: если было `K` числовых ЦОФ и `R` родительских res — добавляется `K · R · 4` колонок (примерно `8 · 2 · 4 = 64` при стартовом наборе).
- На метрики **ожидаем** заметного эффекта (методология §11 называет это «главным выигрышем»). Численная оценка — в commit-сообщении прогона.
- CatBoost native handles NaN — не нужны fallback'и для пограничных случаев.

## Что **не** делаем в этой итерации

- **Полный перебор нормировок (§11).** Методология говорит «каждый дополнительный способ удесятеряет признаки». Сейчас стартуем с консервативного 4 × 2 × K, без mean/std/skew. Расширим, если CatBoost-feature-importance покажет, что эти 64 колонки реально работают.
- **Нелинейные функции влияния.** Монотонные/немонотонные преобразования — после того как relative-фичи продемонстрируют отдачу.
- **Перекрёстные взаимодействия (interactions).** Произведения вида `dist_metro × dist_school` — отложено до §11 второго порядка.
- **Соседи 1/2/3 порядка через `cell_to_disk`.** Технически — добавить ещё один уровень родительства; по методологии полезно. В первой итерации не делаем — родители уже дают пространственный контекст, а disk-агрегаты лучше комбинировать с поли-площадными буферами (блок 3).
- **Символьная регрессия.** Совершенно отдельная история.
- **Per-class агрегаты.** См. выше.

## Открытые вопросы

- **Поведение при крошечных родителях.** На res=8 в редкозаселённых частях буфера (Иннополис, Лаишево) родитель может содержать 1–2 объекта. Тогда `median == F`, `diff_med == 0`, `ratio_med == 1`, `iqr == 0`. Это **корректно** для CatBoost (он сам поймёт через `count_p{R}`, что агрегат не информативен), но может зашуметь линейные модели в White Box-квартете (блок 5). Перепроверим там.
- **Утечка target'а через агрегаты.** Сейчас `relative_feature_columns` — это **признаки**, не таргет. Поэтому утечки нет. Когда добавим относительные агрегаты по `cost_index` (в перспективе по сделочным ценам) — нужен временной/пространственный split при подсчёте агрегатов, иначе модель «увидит» соседей по тому же fold'у. Это будет частью блока 6 (темпоральные срезы).
- **Расширение на ГАР-территории.** Методология говорит про «муниципальный район» / «НП» — это логически родитель, но не H3-родитель. Подключим в блоке 4 (территориальные ЦОФ).
