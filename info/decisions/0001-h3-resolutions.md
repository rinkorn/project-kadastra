# ADR-0001: Выбор рабочих разрешений H3

**Статус:** Superseded by [ADR-0007](0007-kazan-agglomeration-scope.md)
**Дата:** 2026-04-25

## Контекст

Проект оценивает кадастровую стоимость на гексагональной сетке H3. Согласно [project.md](../project.md) и [h3-primer.md](../h3-primer.md), все feature-таблицы имеют ключ `(h3_index, resolution)`, и разрешения должны быть фиксированы заранее. Без выбора нельзя начинать ETL и собирать признаки.

Особенности задачи:

- Объекты оценки — отдельные участки/здания, типичный размер от десятков до тысяч м².
- Признаковая база разная по масштабу: сделки точечные, инфраструктура — линейная (дороги) и точечная (POI), здания — полигональные, а демография/админ-границы — крупные.
- Сервинг отображает гексы на интерактивной карте → нужен уровень, удобный визуально на масштабах город / район / квартал.

## Варианты

### A. Один уровень (например `res=8`)

- Плюсы: простейший ETL и хранение, одна модель.
- Минусы: режется детализация дорогих объектов; в редко-застроенных зонах слишком мало точек на гекс.

### B. Два уровня: `res=8` (основной) + `res=7` (региональный контекст)

- Плюсы: основной уровень для модели, второй — для агрегатных фичей-контекстов («средняя цена района») и обзорной карты.
- Минусы: нужно правило приведения между уровнями (через `h3_to_parent`).

### C. Три уровня: `res=7`, `res=8`, `res=9`

- Плюсы: покрывает обзор / квартал / двор; гибче на карте (zoom-зависимая отрисовка).
- Минусы: ×3 объём ETL, риск рассинхрона версий между уровнями.

## Предложение

**Принять вариант B**: основной `res=8`, контекстный `res=7`.

Обоснование:

- `res=8` (~530 м, ~0.74 км²) — адекватный размер «однородной» зоны для городской недвижимости: внутри одного гекса физика рынка обычно одинаковая. Этот уровень — целевой для модели и feature store.
- `res=7` (~1.4 км, ~5 км²) — естественный родитель для `res=8` (через `cell_to_parent`), используется для контекстных фичей («медианная цена в районе», «плотность сделок в районе») и для обзорной отрисовки на низком зуме.
- `res=9` пока не берём — добавит ×7 объёма данных, при том что для оценки кадастровой стоимости разрешение «двор» избыточно. Поднять можно позже отдельным ADR, если метрики покажут потолок на res=8.

## Последствия

- В `Settings` появляется `h3_resolutions: list[int] = [7, 8]`, `h3_primary_resolution: int = 8`.
- Все ETL-шаги принимают `resolution` параметром; запуск без него запрещён.
- Контекстные фичи на res=7 строятся из агрегатов res=8 через `cell_to_parent` — не из сырых данных дважды.
- Spatial CV режет по res=7-родителям (более крупные кластеры → меньше утечек через соседство).
- Модель обучается и инференсится **только** на res=8.
- Тайлы карты: на низком зуме отдаём res=7, на высоком — res=8.

## Применимо к Татарстану

Регион проекта зафиксирован в [ADR-0002](0002-region-tatarstan.md): Татарстан включает плотный город (Казань), второстепенные центры (Челны, Нижнекамск, Альметьевск) и обширные сельские/полусельские зоны.

- `res=8` остаётся целевым для модели — адекватен для городской и пригородной застройки.
- В сельских гексах будет много пустых ячеек / низкая плотность сделок. Это решается через:
  - фильтрацию обучающей выборки по минимальной плотности (`n_deals >= k`),
  - использование `res=7`-агрегатов как контекстных признаков (разреженность сглаживается на родителе).
- Уход на res=7 как основной не оправдан: для Казани это слишком крупно (~5 км² на ячейку — внутрь попадают разные рыночные зоны).

## Открытые вопросы

- Подтвердить выбор у владельца задачи.
- Нужен ли `res=9` для отображения на максимальном зуме (только визуализация, без участия в модели)?
