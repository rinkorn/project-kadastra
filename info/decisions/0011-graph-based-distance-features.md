# ADR-0011: Дистанционные ЦОФ через OSM-граф вместо евклидова haversine

**Статус:** Accepted
**Дата:** 2026-04-25
**Реализует:** [ADR-0010](0010-methodology-compliance-roadmap.md), пункт 1 дорожной карты трека 2 (методологическое соответствие).
**Источник методологии:** [info/grid-rationale.md](../grid-rationale.md), §7.2–7.3.

## Контекст

До этого ADR `compute_object_metro_features` считал `dist_metro_m` и `dist_entrance_m` как евклидово haversine-расстояние от центра объекта до ближайшей станции метро / входа. Это противоречит методологии:

> [grid-rationale.md §7.2] Граф используется с **путевым временем** (вес ребра — не евклидово расстояние, а с учётом скорости/класса дороги). Ближайший = ближайший по времени, не по геометрии.

Евклидов haversine «прокидывает» через здания, реки и заборы — модель видит две точки на разных берегах Казанки или по разные стороны железной дороги как одинаково близкие к станции метро, хотя реальный путь там в разы длиннее. Для пешеходных ЦОФ (ходьба до метро) это особенно критично.

## Решение

**Заменить евклидов haversine путевым расстоянием по OSM-графу пешеходного движения.**

Текущий шаг — путевое **расстояние** (метры по графу), не время. Для ЦОФ метро это эквивалентно времени с точностью до константы (пешеходная скорость одинакова на всех типах рёбер ≈ 5 км/ч), и потому не вносит ошибки. Когда добавим автомобильные ЦОФ (дист. до больниц, школ за пределом пешей доступности) — введём времеzdoŕные веса по классу дороги отдельным ADR; архитектура порта это поддерживает.

### Архитектура

```text
RoadGraphPort (Protocol)
└── distance_matrix_m(from_coords, to_coords) -> np.ndarray  # (|from|, |to|)

NetworkxRoadGraph (RoadGraphPort)
├── from_edges([(from_coord, to_coord, length_m), ...])     # in-memory build
├── from_parquet(path)                                       # eager load on boot
└── distance_matrix_m: KD-tree snap + per-target Dijkstra
```

Ключевые свойства реализации:

- **Алгоритм.** `single_source_dijkstra_path_length` от каждой target-вершины. Для метро (~11 станций × ~92k зданий) это 11 запусков Dijkstra вместо 1M парных — основа того, что считается за приемлемое время.
- **Снэп.** scipy `cKDTree` ищет ближайший граф-узел за `O(log n)`, а потом возвращаем **истинное haversine**-расстояние от точки запроса до узла как «снэп-цена» (cKDTree сам использует squared-euclidean на (lat, lon)-плоскости — это rank-preserving в малом регионе, но не метрика в метрах).
- **Снэп-цена прибавляется к пути на обоих концах.** То есть `total = snap_from + path_through_graph + snap_to`. Если объект в 50 м от ближайшей улицы, это попадёт в результат.
- **Disconnected.** Если между двумя узлами нет пути — `inf`. Downstream треды сами решают, как с этим жить (модель CatBoost получит NaN после того как мы заменим `inf` на NaN, см. ADR-0010 пункт «target NaN handling»).

### Source граф

Используем `highway=*` из OpenStreetMap, **исключая** `motorway / trunk / motorway_link / trunk_link / construction / proposed`:

- Motorway/trunk пешеходам недоступны и распирают граф искусственно длинными «короткими» путями.
- Construction/proposed — будущие дороги; для сегодняшней модели рынка некорректны.

Покрытие — Казанская агломерация ~30 км буфер ([ADR-0007](0007-kazan-agglomeration-scope.md)). Bbox в скрипте: `55.5..56.05 lat, 48.7..49.55 lon`.

### Pipeline

```text
[Overpass API]
      ↓ scripts/download_walking_network.py
data/raw/osm/kazan_walking_network.json
      ↓ scripts/build_road_graph_artifact.py
data/silver/road_graph/edges.parquet  ← Settings.road_graph_edges_path
      ↓ NetworkxRoadGraph.from_parquet(...)
RoadGraphPort
      ↓ Container.build_road_graph()
BuildObjectFeatures(road_graph=...)
      ↓ compute_object_metro_features(..., road_graph=...)
data/gold/valuation_objects/.../data.parquet  (с graph-distance в dist_metro_m)
```

Edges-table parquet schema: `(from_lat, from_lon, to_lat, to_lon, length_m)` — Float64 каждая. Один way OSM с N узлами разворачивается в N-1 рёбер.

### Library — почему `networkx`

- **Уже стандарт.** Все методологические инструменты Татарстана и подавляющая часть исследовательского OSM-кода под рукой стоит на `networkx`.
- **Достаточно быстр** для N=10⁵ узлов, M=10⁵ рёбер на single-source Dijkstra (~секунда). Для 11 станций × 92k объектов — общее время порядка 30 секунд.
- **Не тащит лишних зависимостей.** Только `networkx` (Pure-Python core). `osmnx` бы тащил `geopandas/scipy/rtree/...`. У нас уже `scipy` (через `shapely`), `polars` для I/O — больше не нужно.
- **Когда понадобится скорость.** Если 30k+ метро-точек в день обращений потребуют sub-second — переедем на `pandana` или `igraph`. Порт не меняется.

### Ограничения первой итерации

- **Только OSM.** Граф ровно настолько полон, насколько полна OSM в Казани. Свежие/частные дворы могут отсутствовать → объект в новом ЖК «сделает крюк» через ближайшую публичную улицу, что в реальности неверно. Лечится регулярным обновлением выгрузки (сейчас снапшот разовый).
- **Расстояние, не время.** Для пешеходных ЦОФ адекватно. Когда подключим автомобильные ЦОФ — отдельный ADR с весами по `highway=*` классу.
- **Без turn-restrictions.** Граф undirected, идём по любому ребру в обе стороны. Для пешехода нормально; для авто — потеряет одностороннее движение.
- **Один граф на регион.** Темпоральных срезов нет (методология §10), будут когда появятся сделки и дата-зависимые расчёты.

## Последствия

- `compute_object_metro_features` сигнатура изменилась: `road_graph: RoadGraphPort` стал обязательным kwarg.
- `BuildObjectFeatures.__init__` принимает и форвардит `road_graph: RoadGraphPort`.
- `Container.build_road_graph()` — eager load из `Settings.road_graph_edges_path` на boot. Если файла нет — `FileNotFoundError` с понятным сообщением (нет графа = нельзя считать фичи метро).
- `Settings.road_graph_edges_path: Path = Path("data/silver/road_graph/edges.parquet")` — новый параметр.
- `requirements`: добавлена зависимость `networkx>=3.4`.
- Два новых CLI-скрипта (`download_walking_network.py`, `build_road_graph_artifact.py`) формируют граф в два шага: download (медленный, network-bound) → build (быстрый, локальный). Каждый идемпотентный.

## Метрики до/после

Ожидание: `dist_metro_m` распределение **сместится вверх** (граф не короче евклидова), особенно для объектов через Казанку или через железную дорогу. `count_stations_1km` для части объектов **уменьшится** (то, что было «1 км по прямой», по графу часто 1.5–2 км).

Эффект на качество модели — отдельный замер при следующем train/infer прогоне; ожидаем небольшого снижения MAPE (методология §11 говорит, что относительные графовые ЦОФ дают «значимое улучшение», но численная оценка для конкретно этого датасета ляжет в коммит-сообщение прогона).

## Открытые вопросы

- **Регулярное обновление графа.** Сейчас снапшот разовый. На прод-цикле нужен джоб «раз в N недель → новый Overpass-fetch → новый артефакт».
- **Темпоральная стабильность.** Когда подключим сделки 3-летней давности (ADR-0010 трек 1), нужны темпоральные срезы графа — иначе старая сделка получит ЦОФ с метро, которого тогда не было.
- **Время vs расстояние.** Введение time-весов по классу дороги — отдельный ADR. Триггер — когда впервые понадобится автомобильная ЦОФ (например, dist до больницы / роддома, где пешком нерелевантно).
- **`road_length_500m`.** Методологически тоже стоит пересчитать через граф (длина рёбер графа в радиусе 500 м), но это уже зональный ЦОФ из §8 grid-rationale — отдельный ADR.
- **Snap-distance threshold.** Сейчас snap всегда добавляется. Если объект сильно удалён от графа (>500 м), это может означать ошибку в координатах объекта, а не реальную «глушь». Возможный шаг — выбрасывать или маркировать такие объекты при rebuild features.
