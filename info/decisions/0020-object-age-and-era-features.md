# ADR-0020: Возраст объекта и эпоха застройки

**Статус:** Accepted (реализован, ждёт замера на retrain)
**Дата:** 2026-04-26
**Реализует:** [info/grid-rationale.md §6](../grid-rationale.md) (Объектные ЦОФ — характеристики самого объекта), [§10](../grid-rationale.md) (Временны́е ЦОФ).
**Опирается на:** существующее поле `year_built` в gold-схеме `valuation_objects`.

## Контекст

В gold-схеме у каждого объекта есть `year_built` — числовое значение года постройки из ЕГРН. Сейчас это поле отдаётся в модель **как есть** (numeric feature). CatBoost с этим разберётся лучше, чем линейная модель — уловит «1960 vs 1990 vs 2020» через nonlinear splits. Но:

1. EBM (White Box) на сырой год даёт отдельный shape function: красивый график, но трактовка «на 1980 цена ниже на X, на 1990 на Y» — менее интуитивная, чем «брежневка на ~30% дешевле новостройки».
2. Naive Linear на сырой год получает почти 0 коэффициент: год монотонной зависимости с ценой не имеет (новостройка дороже сталинки, но межсоветский период тоже сложный).
3. Для интерпретируемости отчёта-валюатора (главная цель ADR-0016) **категория эпохи** — это сигнал, который человек-эксперт распознаёт мгновенно.

Кадастровая практика делит жилой фонд на эпохи: дореволюционная застройка / сталинки / хрущёвки / брежневки / поздний СССР / 90-е / 2000-е / 2010-е / новостройки. Каждая эпоха имеет узнаваемый профиль (планировка, материалы, износ, ремонтопригодность), и рынок их по-разному оценивает.

Чистая feature engineering: всё уже в `year_built`, новые источники не нужны.

## Решение

Из существующего `year_built` производим 4 новые фичи в `compute_object_age_features`:

| фича | формула | смысл |
| --- | --- | --- |
| `age_years` | `current_year − year_built` | абсолютный возраст |
| `age_years_sq` | `age_years²` | нелинейность (новостройка vs ветхий) |
| `era_category` | бинирование по году (см. ниже) | категориальная эпоха |
| `is_new_construction` | `age_years ≤ 5` | бинарный флаг новостройки |

`current_year` берётся из `Settings.current_year_for_age_features` (default 2026), не `datetime.now()` — детерминированность важна для repro между прогонами.

### Эпохи

| код | диапазон год | примерная характеристика |
| --- | --- | --- |
| `pre_revolution` | <1917 | дореволюционная застройка |
| `early_soviet` | 1917–1945 | конструктивизм + ранний соцклассицизм |
| `stalin` | 1946–1956 | сталинский ампир |
| `khrushchev` | 1957–1968 | хрущёвки 5-этажные |
| `brezhnev` | 1969–1980 | брежневки 9–14 этажей |
| `late_soviet` | 1981–1991 | улучшенки, серии 121/97 |
| `90s_transition` | 1992–2000 | переходный период, стихийная застройка |
| `2000s` | 2001–2010 | первый коммерческий цикл |
| `2010s` | 2011–2020 | современный массовый сегмент |
| `new_2020+` | 2021– | новостройки текущего цикла |
| `unknown` | null/0 | year_built отсутствует или нулевой |

Границы выбраны по укоренившемуся «жилищному делению»; границы не критичны — модель устойчива к небольшим сдвигам.

### Применимость по классам

- `apartment`, `house`, `commercial` — все 4 фичи актуальны.
- `landplot` — у участков `year_built` обычно null (год постановки на учёт, не «постройки»). Все фичи будут `unknown`/null. Не отключаем явно — модель сама проигнорирует.

### Что **не** делаем в этой итерации

- **Год последнего капремонта.** Не входит в стандартную ЕГРН-выдачу, есть только в фондах капремонта (отдельный API). Сильный signal, но отдельной ETL-задачей.
- **Динамика рынка во времени.** «Цены строились в каком-то году → в каком году считаем» — это про temporal features per region (см. [grid-rationale §10](../grid-rationale.md)). Это уже не про объект, а про market state, и ADR-0022 (EMISS) частично закрывает.
- **Age × class interactions.** EBM умеет сам учить pairwise при `interactions > 0`; не вмешиваемся вручную.

## Архитектура

```text
gold/valuation_objects/...
  └─ year_built: Int64 (nullable)
            │
            ▼ BuildObjectFeatures.execute(...)
              └─ compute_object_age_features(...)  ← новый шаг
gold/valuation_objects/... (overwrite)
  └─ + age_years, age_years_sq, era_category, is_new_construction
```

Новый модуль:

```text
src/kadastra/etl/object_age_features.py
  _ERA_BINS: list[tuple[int|None, int|None, str]]  # (lo, hi, code)
  def compute_object_age_features(objects: pl.DataFrame, *, current_year: int)
      -> pl.DataFrame
```

Чистая функция, никаких портов. Реализация — три `pl.when().then()` цепочки, нулевая аллокация выше O(N).

### Settings

```python
current_year_for_age_features: int = 2026
```

Раз в год при release повышаем; в run-time не зависим от системного time.

## Эмпирический эффект (гипотеза)

- **apartment**: Δ MAPE −0.3…−1 пп. Эпоха явно различает массовый сегмент (хрущёвка ~50–80 тыс ₽/м²) от премиум (сталинка ~150+, новостройка 200+). EBM получит чёткий shape по `era_category`.
- **house**: Δ −1…−2 пп. House разнообразнее (от ветхого деревянного до коттеджа), эпоха здесь — ёмкая категория.
- **commercial**: Δ ≈ 0…−0.5 пп. Год значит меньше — коммерческое определяется размещением.
- **landplot**: Δ ≈ 0. У участков year_built почти всегда null.

Если для apartment/house эффект окажется ≤ 0.3 пп — значит CatBoost уже всё это извлёк сам из сырого `year_built`, и польза от категоризации в основном **интерпретационная** (для EBM/Naive). Это нормально, ADR держит фичи как self-документирующие, не только как perf-уплотнитель.

## Открытые вопросы

- **Календарный сдвиг.** Если объект построен в 2026 (текущий год) — `age_years = 0` → `is_new_construction = True`. OK.
- **Битые year_built.** В ЕГРН встречаются записи с `year_built = 1` или 1900 у явно новых зданий. Обнаруживается на data-quality этапе ([info/data-sources.md](../data-sources.md) — раздел про NSPD). Если массово — стоит фильтровать. На первой итерации оставляем как есть (модель устойчива).
- **Пригородные/посёлковые house.** Часто у них year_built = null или поставлены на учёт по факту существования без даты. Модель видит `era_category=unknown`, что само по себе — сигнал.
