# ADR-0014: Поли-площадные ЦОФ через OSM-полигоны в буфере

**Статус:** Accepted
**Дата:** 2026-04-25
**Реализует:** [ADR-0010](0010-methodology-compliance-roadmap.md), пункт 3 дорожной карты трека 2 — поли-площадь (мульти-слойный сумм-эффект). Дополняет [ADR-0013](0013-zonal-density-features-multi-radius.md), который закрыл моно-плотность точек.
**Источник методологии:** [info/grid-rationale.md](../grid-rationale.md), §8.

## Контекст

[ADR-0013](0013-zonal-density-features-multi-radius.md) добавил плотность точечных слоёв (`{layer}_within_{R}m`) — это **моно-площадь** в терминах §8. Методология явно говорит, что **поли-площадь объясняла лучше**:

> [grid-rationale.md §8] Несколько слоёв одновременно, доли суммируются (могут дать > 1). Учитывает кумулятивный эффект негатива: «гусиная ферма + завод» ≠ «один негативный объект».

Сейчас модель не видит, что объект окружён водой / промзоной / кладбищем — потому что у нас **точечных** слоёв такого типа нет. ОСМ-полигоны нужны.

## Решение

**Добавить функцию `compute_object_polygon_features` и набор фичей `{layer}_share_{R}m`** — доля площади буфера радиуса R вокруг объекта, покрытая полигонами слоя.

### Слои в стартовом наборе

| Слой | OSM-теги | Полярность | Гипотеза |
| --- | --- | --- | --- |
| `water` | `natural=water`, `waterway=riverbank` | смешанная | Видовой бонус у воды vs затопляемость |
| `park` | `leisure=park`, `landuse=forest`, `leisure=garden` | положительная | Зелёная зона повышает стоимость |
| `industrial` | `landuse=industrial`, `landuse=brownfield` | отрицательная | Промзона снижает стоимость |
| `cemetery` | `landuse=cemetery`, `amenity=grave_yard` | отрицательная | Кладбище — типичный «негативный» слой методологии |

4 слоя × 4 радиуса (100/300/500/800 м из ADR-0013) = **16 новых фичей** + 4 явных «poly-area sum» производных не делаем — CatBoost умеет суммировать через дерево, и возможные веса разные.

### Радиусы

Используем те же `[100, 300, 500, 800]` м, что и в ADR-0013. Поли-площадь и моно-плотность — комплементарные представления окрестности; держать разные радиусы запутает интерпретацию.

### Метрика — UTM-проекция

Расчёт площади в (lat, lon) — некорректен (углы не равноплощадны). Используем `EPSG:32639` (UTM zone 39N) — та же проекция, что в `scripts/build_kazan_agglomeration_boundary.py` для буфера агломерации. На широтах Казани UTM zone 39N даёт искажение площади ≤ 0.1 % — методологически адекватно.

Проекция применяется один раз: при загрузке полигонов и при построении буфера-круга вокруг объекта.

### Архитектура

```text
kadastra/etl/object_polygon_features.py
└── compute_object_polygon_features(
        objects: pl.DataFrame,
        *,
        polygons_by_layer: dict[str, list[BaseGeometry]],   # WGS84 polygons
        radii_m: list[int],
    ) -> pl.DataFrame
```

Контракты:

- `polygons_by_layer[name]` — список `shapely.geometry.base.BaseGeometry` в WGS84. Функция сама проецирует в UTM и строит STRtree для быстрого bbox-поиска.
- `radii_m` — список радиусов в метрах. Колонки `{layer}_share_{R}m: Float64` со значением `[0, 1]`.
- Пустой `polygons_by_layer[name]` → колонка с нулями.
- Пустой `objects` → пустые колонки правильной схемы.

Реализация:
1. Спроецировать lat/lon объектов в UTM 39N.
2. Для каждого слоя: спроецировать полигоны в UTM, построить STRtree.
3. Для каждого объекта построить круговой буфер радиуса `max(radii_m)`, найти кандидатов в STRtree.
4. Для каждого кандидата вычислить пересечение с буфером один раз. Доля = площадь пересечения / площадь буфера.
5. Радиусы меньше `max` обрабатываются через тот же кандидат-набор: `min(intersection_at_R)` пересчитывается с буфером радиуса R.

Для 92k объектов × 4 слоя × 4 радиуса × ~10–50 кандидатов = ~10–50M shapely-intersects. На прогретых геометриях это ~5–15 минут. Если упрётся — добавим `prep()` или сменим на `geopandas.overlay`.

### Source данных

Полигоны достаются из агломерационно-обрезанного PBF (`data/raw/osm/kazan-agg.osm.pbf`, 18 МБ — уже есть от пайплайна зданий) через `osmium tags-filter` + `osmium export -f geojsonseq`. Один скрипт `scripts/extract_osm_polygons.py` гонит 4 слоя в 4 GeoJSON-seq файла:

```text
data/raw/osm/kazan-agg-{layer}.geojsonseq
```

Затем в pipeline эти JSON-seq читаются как WGS84-полигоны и передаются в `compute_object_polygon_features`.

### Settings

```python
poly_area_radii_m: list[int] = [100, 300, 500, 800]
poly_area_layers: dict[str, str] = {
    "water": "data/raw/osm/kazan-agg-water.geojsonseq",
    "park": "data/raw/osm/kazan-agg-park.geojsonseq",
    "industrial": "data/raw/osm/kazan-agg-industrial.geojsonseq",
    "cemetery": "data/raw/osm/kazan-agg-cemetery.geojsonseq",
}
```

## Что **не** делаем

- **Сделать слои-источники в S3.** Промежуточные `kazan-agg-{layer}.geojsonseq` пока локальные — это derived data, идемпотентно собирается из S3-PBF. Положим в S3 если процесс build потребует кросс-машинной воспроизводимости.
- **Веса слоёв «положительный/отрицательный».** Методология их явно перечисляет; мы пока кладём сырые `share` и доверяем CatBoost вытащить знак. White Box модели (блок 5) попросят явные веса — введём отдельным ADR.
- **Сложные слои (свалки, гусиные фермы и т.п.).** Они есть в OSM, но требуют точечной фильтрации тегов и обычно мало представлены в Казани. Расширим, если CatBoost-importance покажет, что 4-слойный набор работает.
- **Видовые места.** Гипотетически — пересечение буфера с «view to water/park». Слишком субъективно для первой итерации; вернёмся в блоке 4 (территориальные).
- **Темпоральная стабильность полигонов.** OSM-снапшот разовый, как и для пешеходного графа (ADR-0011, открытый вопрос). Регулярное обновление — отдельный ops-вопрос.

## Открытые вопросы

- **Хорошо ли работают обнулённые `share`.** Объект в чистом поле (нет ни одной полигональной фичи в 800 м) → все `share = 0`. Это правильно методологически, но в линейных моделях (блок 5) может маскировать «нет данных» от «реально пусто». Tracking по `count_p7/p8` (ADR-0012) даёт ответ — родительский гекс в чистом поле тоже малочислен.
- **Верхняя граница.** Если объект внутри озера — share может быть 1.0. Это правильно, но в отчётах будет выглядеть подозрительно. Audit при первой такой фичи.
- **Расчёт пересечений на холодном кэше.** Первый прогон тяжёлый. Если станет неудобно — переедем на `pyproj.geod` + chunked processing.
- **Расширение `relative_feature_columns` на share-колонки.** Не делаем в этой итерации — сначала измеряем эффект самих share-фичей.
