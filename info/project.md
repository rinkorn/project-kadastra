# Проект kadastra

## О проекте

**kadastra** — оценка кадастровой стоимости с привязкой к гексагональной сетке.

Контур задач:

1. **ETL** — сбор и подготовка данных (объекты недвижимости, сделки, инфраструктура, гео-признаки) с агрегацией в гексагоны.
2. **ML** — обучение моделей оценки кадастровой стоимости.
3. **Сервинг** — REST API с инференсом + веб-карта с раскраской гексагонов по предсказанной стоимости.
4. **Хранение данных** — данные изначально лежат и обрабатываются в гексагональной сетке (H3).

Общий подход (Hexagonal, DI, composition root) — в [dev-rules.md](dev-rules.md). Здесь — конкретное наполнение под kadastra.

```text
src/kadastra/
  domain/           ← Entities: Parcel, HexCell, Valuation, FeatureSet, ModelArtifact
  ports/            ← Интерфейсы (Protocol): RawDataPort, FeatureStorePort,
                       ModelRegistryPort, ValuationPort, TilePort
  adapters/         ← Реализации портов
  usecases/         ← Бизнес-логика:
                       BuildHexFeatures, TrainValuationModel,
                       PredictHexValuation, ServeHexTile
  etl/              ← Шаги пайплайна: extract / transform / load,
                       агрегация point→H3, join полигонов→H3
  ml/               ← Тренировка, валидация, метрики, сериализация моделей
  api/
    routes.py       ← FastAPI REST (JSON, prefix /api)
  web/
    routes.py       ← Web UI роуты (HTML, Jinja2 + htmx)
    templates/      ← Jinja2 шаблоны
    static/         ← Vendored CSS/JS, MapLibre, deck.gl
  config.py         ← pydantic-settings (Settings с feature flags)
  composition_root.py ← Container + create_app
```

### Адаптеры (стартовый набор)

| Порт | Адаптер | Когда |
|------|---------|-------|
| `RawDataPort` | `S3RawData` | Сырые данные из S3 (parquet/csv/geojson) |
| `FeatureStorePort` | `ParquetFeatureStore` | Hex-агрегаты в parquet на S3 |
| `FeatureStorePort` | `PostgisFeatureStore` | Когда понадобятся spatial-запросы |
| `ModelRegistryPort` | `MLflowModelRegistry` | Версии моделей и метрики |
| `ModelRegistryPort` | `LocalModelRegistry` | Файловый fallback для dev |
| `ValuationPort` | `CatBoostValuation` | Инференс обученной модели |
| `TilePort` | `H3TileBuilder` | Подготовка данных для карты (hex → JSON) |

| Флаг | По умолчанию | Контролирует |

## Гексагональная сетка данных

- **H3** — основной индекс пространства. Все агрегаты считаются на фиксированных разрешениях.
- **Базовые разрешения**: уточняются по задаче (ориентир — `res=8` для городской аналитики, `res=9` для околодомовой, `res=7` для регионального обзора). Решение фиксируется в `info/decisions/` при выборе.
- **Сырые геометрии** хранятся как есть (WGS84, EPSG:4326), индексируются в H3 на этапе transform.
- **Признаки** — все feature-таблицы имеют ключ `(h3_index, resolution)` либо `(h3_index, resolution, period)` для временных срезов.
- **Сделки/объекты-точки** → агрегируются в hex (count, mean, median, percentiles, std).
- **Полигоны** (здания, кварталы) → разносятся по hex через intersection / weighted area.
- **Линии** (дороги) → length per hex.

## ETL пайплайн

Принципы:

- **Идемпотентность**: повторный запуск шага на тех же входах даёт тот же выход.
- **Слои данных**: `raw → bronze (нормализованное) → silver (hex-агрегаты) → gold (фичи модели)`.
- **Колоночный формат** между шагами: parquet с явной схемой (pyarrow).
- **Контракты данных**: схема каждого слоя описана типизированно (pydantic / pandera), нарушение — падение пайплайна, а не «как-нибудь починим».
- **Партиционирование**: по `resolution` и `period` (если есть временной срез).
- **Без скрытого состояния**: шаг получает пути входа/выхода через аргументы, не лезет в глобальные переменные.

Оркестратор пока не выбран. На MVP — обычные Python-функции/CLI поверх use case'ов. Когда шагов станет много — Prefect или Dagster (решение фиксируется отдельно).

## ML процесс

- **Воспроизводимость**: фиксированные seed'ы, версии данных через хеш входного parquet.
- **Эксперимент = запись в реестре**: параметры, метрики, версия данных, версия кода, артефакт модели.
- **Валидация**: пространственный CV (split по hex/региону, не случайный) — иначе утечка через соседство.
- **Метрики**: MAE, MAPE, RMSE на стоимость; дополнительно — calibration по перцентилям.
- **Регистр моделей**: модель имеет стадии `staging` / `production`. Сервинг подтягивает только `production`.
- **Объяснимость**: SHAP для топ-фичей при отчётах (опционально, не в инференсе).

## Сервинг и карта

- **API**: `POST /api/predict` — принимает признаки или h3-индекс, возвращает оценку. `GET /api/hex/{h3_index}` — данные ячейки.
- **Тайлы**: серверный эндпоинт отдаёт hex-данные для viewport (bbox + resolution) в формате, понятном `H3HexagonLayer`.
- **Карта**: MapLibre GL JS как базовая карта + deck.gl-оверлей с hex-полигонами, цвет по предсказанной стоимости.
- **Кэш**: предсказания на hex кэшируются (parquet-снимок), пересчёт по расписанию или явному триггеру.

## Доменно-специфичные «нельзя»

- Не передавать «голые» DataFrame через границы модулей без схемы / контракта.
- Не делать random train/test split на пространственных данных — только spatial CV.
- Не хранить модели в git — только через реестр (MLflow / файловый артефактный сторадж).
- Не смешивать разрешения H3 в одной feature-таблице без явного `resolution` в ключе.
